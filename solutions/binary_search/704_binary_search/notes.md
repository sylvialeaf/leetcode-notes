# 704 二分查找 · 知识点笔记

> 按知识点梳理，配合同文件夹 `solution.py` / `problem.md`，以及分类通用模板 `../binary_search.py` 看。

## 1. 读题
- 题意大白话：给一个**升序**整数数组 `nums` 和目标 `target`，在数组里找 `target` 的下标；找不到返回 `-1`。无重复元素。
- 具体例子：`nums=[-1,0,3,5,9,12]`
  - `target=9` → 9 在下标 4 → 返回 `4`
  - `target=2` → 不存在 → 返回 `-1`
- 关键词：**有序**。看到"有序 + 找值"就该想到二分。

## 2. 讲思路（面试就这么讲）
**三句话**：
1. 本质是"在有序数组里找一个数"。
2. 二分查找：每次看中间 `mid`，比较 `nums[mid]` 和 `target`，砍掉不可能的那一半，移动左右指针缩小范围。关键是利用"有序"，每次范围减半。
3. 时间 O(log n)，空间 O(1)，因为每比较一次范围就减半。

**一题三讲**：
- 暴力解：从头到尾一个个比，O(n)。
- 为什么慢：完全没利用"数组有序"这个信息，白白一个个看。
- 怎么优化：有序 → 看中间值就能判断 target 在左半还是右半，一次砍掉一半 → O(log n)。

## 3. 解法步骤 / 组成部分
1. 两个指针圈范围：`left = 0`，`right = len(nums) - 1`（闭区间，整个数组）。
2. 循环条件：`while left <= right`（范围里还有数就继续）。**带等号**，否则漏掉最后一个数。
3. 每轮算 `mid = (left + right) // 2` 后比较，3 种情况：
   - `nums[mid] == target` → `return mid`
   - `nums[mid] < target`（太小）→ 去右半 → `left = mid + 1`
   - `nums[mid] > target`（太大）→ 去左半 → `right = mid - 1`
   - 循环结束没 return → `return -1`

## 4. 复杂度（为什么）
- O(log n) = "n 能对半砍几次才剩 1 个"。16→8→4→2→1 砍 4 次；反过来 2⁴=16，所以次数 = log₂16 = 4。
- 实感：100 万个数约 20 步，10 亿个数约 30 步。对比暴力 O(n) 的百万/十亿次。
- 空间 O(1)：只用了几个指针变量。

## 5. 易错点
- `while` 条件带等号：`left <= right`，漏等号会少查一个数。
- 移动指针要 ±1：`left = mid + 1` / `right = mid - 1`，写成 `left = mid` 可能死循环。
- `mid = (left + right) // 2` 用整除 `//`。

## 6. 面试官追问 Q&A（每题必有）
- **Q：循环条件为什么用 `left <= right` 而不是 `left < right`？**
  A：因为 `right` 初始化成 `len-1`，是闭区间 `[left, right]`，最后一个数也是有效候选。当 `left == right` 时区间里还剩 1 个数，必须再查一次；用 `<` 就会漏掉它。（条件是否带等号，取决于区间是闭还是开。）
- **Q：`mid = (left + right) // 2` 有什么隐患？怎么写更稳？**
  A：在 Java/C++ 里 `left + right` 可能整数溢出，更稳的是 `mid = left + (right - left) // 2`。Python 整数不会溢出，但养成这个习惯没坏处。
- **Q：如果数组有重复元素，要找 target 第一次出现的位置，怎么改？**
  A：普通二分找到任意一个就返回了。要找左边界：命中 target 时**不立刻返回**，记下 `mid` 再 `right = mid - 1` 继续往左缩，直到区间空。（这就是 LC34，见分类通用模板的"模板 B 找左边界"。）
- **Q：如果数组无序，二分还能用吗？**
  A：不能，二分的前提就是有序。无序的话要么先排序 O(n log n) 再二分，要么用哈希表 O(n) 直接查。

## 7. 顺带建立的概念
- 复杂度量级：O(1) 常数 < O(log n) 对数 < O(n) 线性。能"对半砍"的就是 O(log n)。
- 闭区间 vs 开区间：决定 `right` 初值和 `while` 是否带等号、指针怎么动，二分写法的一致性来源于此。
