# 153 寻找旋转排序数组中的最小值 · 知识点笔记

> 按知识点梳理，配合同文件夹 `solution.py` / `problem.md`，以及分类通用模板 `../binary_search.py` 看。
> 和 33 是"同源题"（都在旋转数组上二分），但用的是**另一套二分模板**，对比见第 4 节。

## 1. 读题
- 题意大白话：旋转过的升序数组（元素互不相同），找**整个数组里最小的那个数**，返回它的值。要求 O(log n)（不准 `min()` 扫一遍）。
- 例子：
  - `[4,5,6,7,0,1,2]` → 最小值 `0`
  - `[3,4,5,1,2]` → 最小值 `1`
  - `[1,2,3,4,5]`（没旋转）→ 最小值 `1`（就是第一个）
- 画面感：最小值 = **"断崖"的底部 = 旋转点**。本质用二分找这个断崖。

## 2. 讲思路（面试就这么讲）
**三句话**：
1. 本质是"在旋转数组里找断崖底部（最小值/旋转点）"。
2. 比 `nums[mid]` 与 `nums[right]` 判断断崖在哪半边，往含最小值的那半缩，缩到只剩一个数就是答案。
3. 时间 O(log n)，空间 O(1)。

**一题三讲**：
- 暴力：`min(nums)` 或扫一遍，O(n)。
- 为什么不行：题目要 O(log n)。
- 怎么优化：利用"两段各自升序、最小值在断崖底"的结构，二分逼近断崖。

## 3. 核心判断（为什么拿 mid 和 right 比）
最小值在右段底部，所以拿 mid 跟**右端点**比最能看出断崖在哪边：
- `nums[mid] > nums[right]`：mid 居然比右端还大 → mid..right 之间**跨了断崖** → 最小值在 **mid 右边**，且 **mid 一定不是最小值** → `left = mid + 1`（排除 mid）。
- `nums[mid] < nums[right]`：mid..right **没跨断崖、是升序的** → 最小值在 **mid 左边，且 mid 本身可能就是最小值** → `right = mid`（保留 mid）。

## 4. ⭐ 与前 4 题不同的二分模板（最重要的收获）
153 用的是**"缩到单个幸存者"**模板，和 704/35/34/33 的"找特定值/边界"模板不同：

| | 704/35/34/33（找值 / 找边界） | 153（缩到单个幸存者） |
|---|---|---|
| 循环条件 | `while left <= right` | `while left < right` |
| 缩边写法 | 两边都 `mid ± 1` | 一边 `left = mid + 1`，一边 `right = mid` |
| 结束方式 | 命中就 return / 指针交叉 left>right | `left == right` 撞到一起 |
| 返回 | `mid` / `left` / `-1` | `nums[left]`（幸存者） |

### 新点 1：`right = mid`，不是 `mid - 1`
当"最小值在 mid 左边、**含 mid 自己**"时，mid 可能就是答案，**不能排除它**，所以保留 → `right = mid`。
（对比：另一支"mid 铁定不是答案"，可以排除 → `left = mid + 1`。）

### 新点 2：`while left < right`，不是 `<=`
不是"找到就 return"，而是**不停缩范围直到 left、right 撞到一起、只剩一个数**，那个幸存者就是最小值。所以 `left < right`，撞上（`left==right`）即停，返回 `nums[left]`。

> ⚠️ **新点1 和 新点2 是一套的**：`right = mid`（不带 -1）必须配 `while left < right`。
> 若配 `<=`，当 `left==right==mid` 时区间缩不动 → **死循环**。记住这对搭配。

## 5. 解法步骤
1. `left=0, right=len-1`。
2. `while left < right`：算 `mid`。
   - `nums[mid] > nums[right]` → `left = mid + 1`
   - 否则 → `right = mid`
3. 跳出时 `left == right` → `return nums[left]`。

## 6. 复杂度
- 时间 O(log n)：每轮砍一半。空间 O(1)。

## 7. 易错点
- ⚠️ `right = mid` **不能写成 `mid - 1`**：会把可能是答案的 mid 弄丢。
- ⚠️ 循环必须 `left < right`：配 `right = mid` 才不死循环；写成 `<=` 会死循环。
- ⚠️ 比较对象是 `nums[right]`（右端点），不是 `nums[mid+1]` 或 `nums[left]`。用 `nums[left]` 比在"没旋转/已升序"时会判错。
- 返回的是**值** `nums[left]`，不是下标 `left`（题目要最小元素的值）。

## 8. 面试官追问 Q&A（每题必有）
- **Q：为什么和 `nums[right]` 比，不和 `nums[left]` 比？**
  A：和 `nums[left]` 比在"数组没旋转（整体升序）"时会出错——此时任意 mid 都 `>= nums[left]`，会一直往右丢，丢掉真正在最左的最小值。和 `nums[right]` 比则天然兼容"没旋转"（此时 `nums[mid] < nums[right]` 恒成立，一路 `right=mid` 收敛到下标 0）。
- **Q：`right = mid` 为什么不会死循环？**
  A：因为配了 `while left < right`。当区间剩两个数时 `mid` 取到左边那个，要么 `left=mid+1` 让两者相等跳出，要么 `right=mid` 也让区间收缩到 1。只要 `mid` 严格小于 `right`（整除取下界保证了这点），`right=mid` 每次都真的缩小区间。
- **Q：如果有重复元素（如 [3,3,1,3]）怎么办？**
  A：`nums[mid]==nums[right]` 时无法判断断崖在哪边，需要 `right -= 1` 收一格再判，最坏退化 O(n)。那是 LeetCode 154 题。本题元素互不相同，不会遇到。
- **Q：找到最小值的下标（旋转点）有什么用？**
  A：旋转点把数组分成两段有序区间，知道它就能把"旋转数组搜索"退化成"普通二分"（33 的两次二分解法就是先找它）。最小值下标 = 旋转次数。

## 9. 顺带建立的概念
- **二分有两套常见模板**，按"答案能不能被排除"选：
  - 当前 mid 判断后**一定能排除**（要么命中 return、要么不是答案）→ 用 `<=` + `mid±1`（704/33）。
  - 当前 mid **可能就是答案、不能排除**（要保留它继续逼近）→ 用 `< ` + 一边 `=mid`（153 找最小值、求左边界的另一种写法）。
- 选错搭配最常见的后果就是**死循环**，所以"`=mid` 配 `<`、`mid±1` 配 `<=`"要成对记。
